## Project Structure

A typical Spinai project has the following structure:

```bash
my-spinai-project/
├── actions/              # Your AI actions
│   ├── search.ts
│   └── process.ts
├── orchestrator/         # Orchestrator configuration
│   └── index.ts
├── spinai.config.ts     # Spinai configuration
└── package.json
```

## Creating Actions

Actions are the building blocks of your AI agent. Each action should be in its own file:

```typescript
import type { ActionModule } from "spinai";

export const config = {
  id: "myAction",
  dependsOn: ["otherAction"], // Optional dependencies
  metadata: {
    description: "Performs a specific task",
  },
};

export async function run(context) {
  // Action implementation
  return result;
}
```

### Action Context

The context object provides access to:

- `request.input`: User input string
- `request.metadata`: Additional request metadata
- `store`: Shared state between actions
- `actions`: Access to other registered actions

## Configuring Orchestrators

Orchestrators manage the flow of your AI agent. The OpenAI orchestrator is included by default:

```typescript
import { createOpenAIOrchestrator } from "spinai";

export default createOpenAIOrchestrator({
  apiKey: process.env.OPENAI_API_KEY,
  model: "gpt-4-turbo-preview", // Optional, defaults to gpt-4-turbo-preview
  temperature: 0.7, // Optional
  systemPrompt: `Your system prompt here...`,
});
```

## Dependency Management

Spinai automatically handles action dependencies:

```typescript
export const config = {
  id: "processData",
  dependsOn: ["fetchData", "validateData"],
  allowRerun: false, // Optional, prevents multiple runs
};
```

## Debugging

Use the built-in logger for debugging:

```typescript
import { log } from "spinai";

export async function run(context) {
  log("Processing request", { level: "debug", data: context.request });
  // Action implementation
}
```

Log levels:

- `debug`: Detailed debugging information
- `info`: General information (default)
- `error`: Error conditions

## Testing

We recommend testing your actions in isolation:

```typescript
import { ActionContext } from "spinai";

describe("myAction", () => {
  it("should process input correctly", async () => {
    const context: ActionContext = {
      request: { input: "test" },
      store: {},
      actions: {},
    };

    const result = await myAction.run(context);
    expect(result).toBeDefined();
  });
});
```

## Best Practices

1. Keep actions focused and single-purpose
2. Use descriptive action IDs and metadata
3. Handle errors gracefully
4. Use dependency management for complex workflows
5. Log important information for debugging
